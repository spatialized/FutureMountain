# Future Mountain Database

This folder is the central reference point for Future Mountain database structure and database-facing documentation.

It does not contain live MySQL database files. Local and server databases are managed through MySQL Server/MySQL Workbench. Files in this folder are source-controlled references that help developers recreate, inspect, compare, or update the database contract.

## Current Contents

```text

    DataDictionary.md
  Diagrams/

```

## Source Of Truth

Use this folder as the project-level source of truth for database structure:

- `Schema/` contains exported SQL schema snapshots.
- `Docs/DataDictionary.md` explains tables and fields in human-readable form.
- `Diagrams/` contains the MySQL Workbench EER model and exported diagram.

Importer-specific Entity Framework migration files currently remain in:

```text
RHESSYs_Data_Importer/RHESSYs_Data_Importer/Migrations/
```

Those files are generated by the importer project and are useful for applying schema changes through EF Core. They should be treated as implementation artifacts for the importer, while this `Database/` folder remains the project-wide reference for the expected database contract.

## Schema Files

The schema files are intended to be structure-only exports from MySQL Workbench or equivalent tooling.

Suggested naming:

```text
BigCreekV1_schema.sql
CentralCoastV2_schema.sql
```

When a schema changes, update the relevant schema export and the data dictionary in the same commit.

## Migrations

This project imports scenario data rarely and usually in large batches. Because of that, a full hand-authored migration system is not currently a high priority.

For now:

- Keep EF Core migrations with the importer project.
- Keep exported schema snapshots in `Database/Schema/`.
- Add a project-level `Database/Migrations/` folder only if we start needing hand-written, scenario-independent SQL changes that should be reviewed outside the importer codebase.

If a future migration folder is added, it should contain intentional SQL changes such as:

```text
Database/Migrations/
  001_add_central_coast_fields.sql
  002_add_import_run_table.sql
```

## Seed Data

No seed-data folder is required right now.

Future seed files may be useful if the API/runtime gains small shared reference tables, such as:

- scenario definitions
- warming/climate-case definitions
- variable definitions
- display labels and units

Do not commit full RHESSys scenario datasets here unless they are intentionally tiny samples. Full imports should remain in MySQL databases or external data storage.

## Updating The Database Reference

When the database changes:

1. Apply or generate the change through the importer project, MySQL Workbench, or a reviewed SQL script.
2. Export the updated structure-only schema into `Database/Schema/`.
3. Update `Database/Docs/DataDictionary.md`.
4. Update related importer/API/Unity docs if the contract changed.
5. Smoke test the importer and API against a local or staging database before touching production data.
